# Assign Engine Spec

Date: 2026-05-03

This spec defines the intended behavior of the Assign task engine. It is not a layout mockup. Use it as the product and engineering contract before changing Assign, Tasks, task requests, approvals, or related RPCs.

The goal is to keep the engine aligned with the project cornerstones: clean and DRY code, performance-conscious UI, accessibility, and secure backend practices.

## Core Principle

Assign is the system of record for work that one person gives to another person or to the studio pool.

The UI may show that work in different places, but the behavioral model should stay consistent:

- Admins assign, edit, delete, approve, reject, and manage templates.
- Members complete assigned work and request changes when they should not directly mutate the record.
- Logs record what happened.
- Backend RPCs enforce role and team boundaries.
- The frontend reflects confirmed backend results instead of pretending a mutation succeeded.

## Core Records

The current task system centers around these records:

- `assigned_tasks`: the real task rows users see and complete.
- `task_assignment_batches`: grouped admin assignment actions.
- `assignment_recipients`: recipients for an assignment batch.
- `assignment_notifications`: notifications generated by assignments and requests.
- `task_requests`: member requests for task creation, deletion, transfer, and edit-style flows.
- `task_reassign_requests`: transfer-specific request support where still used.

## Task Scopes

### Member Task

A member task has a specific assignee.

Expected behavior:

- It appears in that member's task surfaces.
- The assignee can complete it.
- Admins can manage it.
- Other members should not mutate it directly.

### Studio Task

A studio task belongs to the shared studio pool.

Expected behavior:

- It appears in shared studio/team task surfaces.
- Any authenticated team member may complete it when the RPC allows that behavior.
- Admins can manage it.
- The UI should make clear that it is studio-pool work, not a private member task.

## Admin Capabilities

Admins should be able to:

- create a member task
- create a studio task
- apply templates
- edit task title, description, flywheel stage, due date, scope, and assignee where supported
- delete selected tasks
- delete a single task from a row action
- approve or reject member requests
- see assignment and approval history

Admin mutations must be team-scoped in the backend. A hidden UI control is not enough.

## Member Capabilities

Members should be able to:

- view their own assigned tasks
- view studio tasks where appropriate
- complete or uncomplete tasks they are allowed to act on
- request a new task when they need work assigned
- request deletion when a task is wrong, obsolete, duplicated, or no longer applicable
- request transfer when another person should own the task
- request edit when task details need correction but the member should not directly change the record

Members should not be able to directly edit or delete admin-owned task rows unless a future product decision explicitly changes that boundary and the backend enforces it.

## Request Types

The request system should present one family of flows rather than separate one-off designs.

### Create Request

Purpose: member asks an admin to create a task.

Expected behavior:

- Member submits title and optional details.
- Admin sees the request in the approval queue.
- Approval creates the task.
- Rejection records the decision and optional note.

### Delete Request

Purpose: member asks an admin to remove a task.

Expected behavior:

- Member chooses a target task and optional reason.
- Admin sees the exact task, request kind, requester, and reason.
- Approval deletes or archives according to the current backend action.
- The UI only removes the task after the backend confirms the affected ID.
- Partial or zero-delete results must be visible, not hidden by optimistic cache removal.

### Transfer Request

Purpose: member asks that ownership move to someone else.

Expected behavior:

- Member chooses a target task and proposed recipient where supported.
- Admin sees current owner, proposed owner, and reason.
- Approval updates ownership through an RPC.
- Related task caches and logs are invalidated after confirmation.

### Edit Request

Purpose: member asks an admin to change task details.

Expected behavior:

- Member chooses a target task.
- Member provides proposed edits and optional reason.
- Admin sees current values and proposed values side by side.
- Approval applies changes through an RPC.
- Rejection keeps the original task unchanged.

Implementation note: if the backend currently raises unsupported for `kind = 'edit'`, the UI should not pretend edit requests are complete. Build the UI only when the RPC path is ready or make the unsupported state explicit.

## Logs

Logs should answer two separate questions.

### Assign Log

Question: what work was assigned or changed?

Good entries:

- task assigned
- task edited by admin
- task deleted or archived by admin
- template applied
- studio task created

### Approval Log

Question: what requests did admins decide?

Good entries:

- create request approved/rejected
- delete request approved/rejected
- transfer request approved/rejected
- edit request approved/rejected

Logs are history surfaces. They should not be the primary place to operate on active tasks.

## UI Contract

Task action placement should be consistent:

- Primary task actions belong near the top of the task widget or task detail surface.
- Per-row actions may appear on the row when the action clearly belongs to that one task.
- Selection mode should show a clear bulk action bar.
- Bulk delete must show how many tasks are selected.
- Empty states should say what the user can do next.

Accessibility requirements:

- Use real buttons for actions.
- Icon-only buttons need accessible labels.
- Destructive actions need clear labels and confirmation where appropriate.
- Selection state cannot be color-only.
- Error and partial-success feedback should be visible text.

## Backend Contract

All task mutations should go through RPCs or RLS-safe query paths.

Rules:

- Admin/owner rights are checked in the database.
- SECURITY DEFINER RPCs set `search_path TO 'public'`.
- Grants are intentional and narrow.
- Mutations are team-scoped.
- Delete/edit/transfer RPCs return affected IDs or structured counts.
- The frontend invalidates all affected query keys after a mutation.
- Legacy null `team_id` or fallback-team records should be treated as drift and repaired by migration/RPC, not by UI guesswork.

## Cache + Realtime Contract

After a task mutation, affected surfaces should update without hard refresh.

At minimum, mutation success should invalidate:

- selected member tasks
- global assigned task lists
- admin task lists
- team task lists
- studio task lists
- approval/request queues where relevant
- logs where relevant

Optimistic UI is allowed only when rollback is clear and the backend result is checked. For destructive actions, prefer confirmed removal based on returned IDs.

## Test Data + Archive Contract

Test data should become first-class instead of relying on one-off SQL cleanup.

Expected direction:

- Add `is_test_data` where useful.
- Prefer `archived_at` for business records that should leave active workflows but remain auditable.
- Reserve hard delete for test data, corrupted rows, or rows with no audit value.
- Future Settings > Data Management should show counts before any bulk action.

## Preview Acceptance

Before merging Assign/task engine changes:

- Use `docs/pr-acceptance-checklist.md`.
- Verify the changed flow in Vercel preview.
- If migrations exist, apply and verify Supabase separately from Vercel.
- Confirm member/admin role separation.
- Confirm partial and failure states are visible.
- Confirm docs are updated in the same PR.

## Suggested Implementation Order

1. Finish member-side edit requests if the backend path is ready.
2. Stabilize Assign delete/edit/transfer feedback and logs.
3. Add test-data/archive support for assigned tasks.
4. Add Settings > Data Management for safe cleanup.
5. Revisit Assign page layout only after the engine contract is stable.

Do not let layout redesigns redefine task behavior. Behavior comes first; layout presents it.
